# Домашнее задание №4 - Notification Platform

## Для начала...
Зафиксируем ключевые решения:

- Транзакционные уведомления (подтверждение перевода, списание средств) нельзя отключить
- Сервисные и маркетинговые пользователь может отключить по желанию
- Push доставляется за <= 3 сек (P99, best-effort), SMS гарантированно за <= 30 сек если push не дошел (P99)
- Failover идет по цепочке push -> SMS -> email

---

## 1. Функциональные требования

**FR-1.** Система принимает уведомления от всех внутренних сервисов через единый API и доставляет их пользователям. Каждый сервис перестает слать уведомления самостоятельно.

**FR-2.** Система различает три типа: транзакционные (обязательные), сервисные и маркетинговые. Тип определяет приоритет обработки и то, может ли пользователь отказаться от таких уведомлений.

**FR-3.** Пользователь через настройки приложения может отключить сервисные и/или маркетинговые уведомления, а также выбрать предпочтительный канал доставки.

**FR-4.** При недоступности основного канала система автоматически переключается на резервный по цепочке push -> SMS -> email. Пользователь ничего не делает вручную.

**FR-5.** Система поддерживает массовую рассылку маркетинговых кампаний на аудиторию до 1 млн пользователей с управляемой скоростью, не перегружая провайдеров.

**FR-6.** Повторная передача одного и того же события не приводит к дублированию уведомления - дедупликация на уровне платформы.

**FR-7.** Отправитель может получить статус уведомления: принято / отправлено в канал / доставлено / ошибка.

---

## 2. Нефункциональные требования

**NFR-1. Задержка (Latency)**
- Push: P99 <= 3 сек от события
- SMS failover: P99 <= 30 сек от события, если push не доставлен
- Email: резервный канал, <= 5 мин

**NFR-2. Пропускная способность (Throughput)**
- Пиковая обработка: >= 50 000 уведомлений/сек
- Маркетинговые кампании: 1 млн уведомлений за 30 минут в batch-режиме

**NFR-3. Надежность (Reliability)**
- Availability платформы: >= 99.9%
- Потеря транзакционных уведомлений: 0 (не допускается)
- При временной недоступности провайдера - ретраи до успеха или исчерпания всех каналов

**NFR-4. Масштабируемость (Scalability)**
Горизонтальное масштабирование без изменения архитектуры при росте DAU с 3 млн до 10 млн.

**NFR-5. Идемпотентность**
Повторная отправка события с тем же идентификатором не создает дубликат. Гарантируется платформой независимо от поведения вызывающего сервиса.

**NFR-6. Наблюдаемость (Observability)**
- Полный жизненный цикл каждого уведомления трассируется
- Метрики в реальном времени: delivery rate, failover rate, latency по каналам
- Алерт при падении delivery rate ниже порогового значения

**NFR-7. Безопасность**
- Персональные данные (телефон, email) не логируются в plaintext
- Доступ к API платформы только для авторизованных внутренних сервисов (mTLS)
- Хранение персональных данных в РФ (152-ФЗ)

## 3. Архитектурно значимые требования (ASR)

### ASR-1. Гарантированная доставка транзакционных уведомлений [Приоритет: Critical]

Связанные требования: FR-4, FR-6, NFR-1, NFR-3

Это самое тяжелое требование с архитектурной точки зрения. Нельзя просто отправить запрос к провайдеру и забыть - нужно персистировать состояние до получения подтверждения. Это тянет за собой необходимость явной state machine доставки, хранилища с ACID-гарантиями, ретраев с защитой от дублирования и dead letter queue для случаев полного отказа всех каналов. Фактически этот ASR формирует всю подсистему delivery engine.

### ASR-2. Изоляция нагрузки по типам уведомлений [Приоритет: High]

Связанные требования: FR-5, NFR-2, NFR-4

Нагрузка принципиально неоднородна: транзакционные - равномерные ~600/сек в пике, маркетинговые - spike до 50 000/сек при запуске кампании. Если обрабатывать их вместе, кампания на 1 млн пользователей просто заблокирует очередь и транзакционные не уложатся в SLA. Значит нужны раздельные очереди с приоритетами и независимое масштабирование воркеров.

### ASR-3. Устойчивость к отказам внешних провайдеров [Приоритет: High]

Связанные требования: FR-4, NFR-1, NFR-3

Внешние SMS/email провайдеры нестабильны по условию задачи. Синхронный вызов провайдера в критическом пути - это гарантированная проблема: один таймаут на 10-30 сек и SLA нарушен. Значит взаимодействие с провайдерами должно быть асинхронным, с circuit breaker на каждом провайдере и логикой автоматического failover. Желательно иметь несколько провайдеров на один канал (особенно для SMS).

## 4. Ключевые архитектурные вопросы

**AQ-1. Как обеспечить exactly-once семантику при failover?**

Порождают: ASR-1, FR-6, NFR-5

Проблема в том, что при failover система начинает попытку через SMS, не получив подтверждения от push. В этот момент push может всё-таки доставиться с задержкой - и пользователь получит два уведомления. Нужно решить, в какой момент считать уведомление доставленным и прекращать попытки. Ответ влияет на то, как устроена delivery state machine и как именно мы обрабатываем входящие delivery receipt от провайдеров.

**AQ-2. Как изолировать транзакционные уведомления от маркетинговых по ресурсам?**

Порождают: ASR-2, NFR-2

Отдельные топики с приоритетом, физически отдельные кластеры или rate limiting на уровне приема - у каждого варианта своя стоимость и операционная сложность. Самое простое - раздельные топики с отдельными consumer group, но это не спасет если брокер один и у него кончаются ресурсы. Самое надежное - физическая изоляция, но это дорого.

**AQ-3. Как обрабатывать delivery receipt от провайдеров?**

Порождают: ASR-3, NFR-6, ASR-1

SMS-провайдеры подтверждают доставку через webhook с задержкой в секунды-минуты. До получения receipt непонятно, доставлено ли сообщение. Запускать failover по таймауту или ждать явного failure? Слишком короткий таймаут - лишние SMS и деньги. Слишком длинный - нарушение SLA 30 сек. Нужна эмпирическая калибровка на реальных данных.

## 5. Архитектурные последствия ASR

**ASR-1 -> Гарантированная доставка:**

- Надежное хранилище состояний - каждое уведомление записывается в БД до отправки (outbox pattern)
- Явная state machine: PENDING -> SENDING -> DELIVERED / FAILED -> FAILOVER -> ...
- Идемпотентные операции - все операции доставки имеют idempotency key
- Разделение приема и доставки - сервис-отправитель получает 202 Accepted, не ждет факта доставки
- Dead letter queue - уведомления без доставки через все каналы попадают в DLQ с алертом

**ASR-2 -> Изоляция нагрузки:**

- Отдельные топики/очереди для транзакционных, сервисных и маркетинговых
- Воркеры для каждого типа масштабируются независимо
- Rate limiter на исходящем трафике к провайдерам
- Маркетинговые кампании отправляются пачками с управляемой скоростью

**ASR-3 -> Устойчивость к провайдерам:**

- Circuit breaker на каждом адаптере провайдера
- Несколько провайдеров на канал SMS (два провайдера, round-robin + failover внутри канала)
- Retry с экспоненциальным backoff + jitter
- Таймаут-based failover при отсутствии delivery receipt

## 6. Архитектурные решения, которые не подходят

**Решение 1: Синхронная цепочка вызовов провайдеров в критическом пути**

При получении транзакционного уведомления сервис синхронно вызывает push-провайдера, ждет ответа, при ошибке идет к SMS и так далее — всё в рамках одного запроса.

Нарушает ASR-3 и ASR-1.

Таймаут одного провайдера (5-30 сек) блокирует поток и нарушает SLA <= 3 сек. При падении сервиса в середине цепочки состояние теряется и непонятно, было ли уведомление отправлено. Не масштабируется: каждый запрос держит thread до конца цепочки.

**Решение 2: Единая очередь для всех типов уведомлений**

Все уведомления — транзакционные, сервисные, маркетинговые — попадают в одну общую очередь и обрабатываются одним пулом воркеров в порядке FIFO.

Нарушает ASR-2.

Маркетинговая кампания на 1 млн пользователей заливает очередь и блокирует транзакционные. Нельзя применить разные политики ретраев и TTL для разных типов. SLA транзакционных (push <= 3 сек) нарушается при высокой нагрузке кампании.

## 7. Неопределенности и архитектурные риски

**Неопределенность 1: Надежность delivery receipt от push-провайдеров**

FCM/APNs не гарантируют receipt в реальном времени. Push может доставиться с задержкой (устройство оффлайн), подтверждение может прийти через минуты. Непонятно, через какой таймаут считать push "не доставленным" и запускать SMS-failover.

Чтобы это выяснить, нужно нагрузочное тестирование с реальными FCM/APNs на тестовой аудитории - замерить P50/P95/P99 времени receipt и на основе этих данных выбрать таймаут. Дополнительно стоит мониторить процент "silent failures" - когда уведомление отправлено, но receipt не пришёл никогда.

**Неопределенность 2: Одновременный запуск нескольких крупных кампаний**

ТЗ допускает кампании до 1 млн пользователей, но не говорит сколько их может быть одновременно. Если одновременно стартуют 3-5 кампаний, пиковая нагрузка в разы превышает расчетную, а rate limiter провайдеров срежет пропускную способность.

Здесь нужно провести интервью с маркетинговой командой и выяснить реальные сценарии использования. После этого - ввести ограничение на количество одновременно активных кампаний на уровне платформы и проверить нагрузочным тестом с 3 параллельными кампаниями по 1 млн.

## 8. RFC: Гарантированная доставка критичных уведомлений с кросс-канальным failover

---

### Контекст и масштаб

**Пользователи:**
- MAU: 10 млн, DAU: 3 млн, Peak Concurrent: 300 000

**Расчет нагрузки по уведомлениям:**

| Тип | В день | Среднее/сек | Пик/сек |
|-----|--------|-------------|---------|
| Транзакционные | 6 млн (3M DAU x 2) | 70 | ~600 |
| Сервисные | 9 млн (3M x 3) | 104 | ~800 |
| Маркетинговые | 15 млн (3M x 5) | 174 | ~50 000 (кампании) |
| Итого | 30 млн | 350 | ~50 000+ |

Push-запросов фактически больше - большинство транзакционных доставляются через push без failover. SMS отправляется только при failover, ожидаем ~5-10% транзакционных = 300-600 тыс. SMS/день. Это важно с точки зрения стоимости.

**Требования к подсистеме:**
- Доставка транзакционного уведомления хотя бы через один канал
- Автоматический failover push -> SMS -> email
- Дедупликация при ретраях и failover
- Полный жизненный цикл каждого уведомления отслеживается
- Push P99 <= 3 сек, SMS failover P99 <= 30 сек, availability >= 99.9%

### Вариант A: Outbox + Kafka Choreography

В основе варианта лежит outbox pattern: каждый внутренний сервис записывает событие в свою БД атомарно вместе с бизнес-данными. Outbox poller вычитывает новые записи и публикует их в Kafka. Delivery Engine подписывается на топики и управляет доставкой. Центрального оркестратора нет — участники взаимодействуют через события.

#### Как выполняет ASR

**ASR-1:** Outbox pattern гарантирует - событие не потеряется даже при падении сервиса, потому что оно записано в БД атомарно с бизнес-транзакцией. Delivery DB хранит состояние до подтверждения. DLQ при полном отказе всех каналов.

**ASR-2:** Отдельные Kafka-топики по типам: `txn.notifications` с высоким приоритетом и отдельным consumer group. Маркетинговые воркеры не влияют на транзакционные даже при пиковой нагрузке кампании.

**ASR-3:** Circuit breaker на каждом адаптере. При ошибке/таймауте - автоматический failover через state machine. Два SMS-провайдера на уровне SMS Adapter с round-robin и failover внутри канала.

#### Технологии

- Брокер: Apache Kafka (топики по типам, репликация x3)
- CDC / Outbox Poller: Debezium (Kafka Connect)
- Delivery Engine: Go
- Delivery DB: PostgreSQL (ACID, надежное хранилище состояний)
- Push: FCM + APNs
- SMS: МТС + Beeline через единый адаптер
- Email: SendGrid
- Observability: Prometheus + Grafana + OpenTelemetry

### Вариант B: Saga Orchestration

В этом варианте вводится центральный Notification Orchestrator, который получает запрос и ведет всю цепочку доставки как сагу. Он хранит состояние, явно управляет шагами и является единой точкой принятия решений о failover. Внутренние сервисы вызывают Orchestrator напрямую через API.

#### Как выполняет ASR

**ASR-1:** Сага персистируется в БД на каждом шаге. При рестарте Orchestrator восстанавливает незавершенные саги. Scheduler обнаруживает зависшие саги и инициирует failover.

**ASR-2:** Раздельные очереди в RabbitMQ с приоритетами. Воркеры масштабируются независимо. Маркетинговые саги запускаются через отдельный endpoint с throttling.

**ASR-3:** Воркеры изолированы от Orchestrator. Отказ провайдера = failure callback -> Orchestrator решает о failover.

#### Технологии

- Orchestrator: Java / Spring Boot
- Saga DB: PostgreSQL
- Scheduler: Quartz (встроен в Spring)
- Message Queue: RabbitMQ (priority queues, проще в ops чем Kafka для task queues)
- Воркеры: Go
- Observability: Prometheus + Grafana + OpenTelemetry

### Сравнение и выбор

| Критерий | Вариант A (Outbox + Kafka) | Вариант B (Saga Orchestration) |
|----------|--------------------------|-------------------------------|
| Гарантия доставки | Высокая (outbox - транзакционная запись) | Высокая (сага персистируется) |
| Latency транзакционных | Ниже (нет синхронного API в критическом пути) | Чуть выше (HTTP-вызов Orchestrator) |
| Наблюдаемость | Сложнее (состояние распределено) | Проще (Orchestrator - единая точка) |
| Операционная сложность | Выше (Kafka + Debezium) | Ниже (меньше движущихся частей) |
| Масштабируемость | Лучше (Kafka горизонтально) | Хуже (Orchestrator - потенциальное узкое место) |
| Дебаггинг failover | Сложнее (события из нескольких сервисов) | Проще (вся логика в одном месте) |
| Coupling | Слабое (choreography) | Сильнее (сервисы зависят от API Orchestrator) |

**Выбор: Вариант A**

При 300 000 peak concurrent users и пике ~600 транзакционных/сек Orchestrator в варианте B становится узким местом: любой синхронный вызов добавляет latency и создает single point of failure. Kafka горизонтально масштабируется и позволяет изолировать транзакционные уведомления на уровне топиков с гарантией порядка.

Второй ключевой аргумент - outbox pattern решает проблему двойной записи (бизнес-событие + событие уведомления) атомарно. В банковском контексте важна атомарность — либо перевод случился и уведомление о нём будет точно отправлено, либо ни то ни другое.

**Ключевые компромиссы и ограничения:**

1. Операционная сложность Kafka + Debezium - требует более зрелой DevOps культуры. Debezium нужно настраивать и мониторить отдельно.

2. Observability сложнее - состояние доставки распределено между несколькими сервисами. Решается сквозным trace ID через OpenTelemetry и Delivery DB как источником правды по статусам.

3. Choreography сложнее отлаживать - при сбое нужно собирать картину из нескольких топиков и логов. Митигируется структурированным логированием с correlation ID.

4. Таймаут failover требует калибровки - начать с 5 сек для push и скорректировать после анализа реальных данных receipt от FCM/APNs (см. Неопределенность 1).

**Дополнительные решения подсистемы:**

Для идемпотентности каждое уведомление получает `idempotency_key = SHA256(source_service + event_id + user_id)`. При повторном событии Delivery Engine находит ключ в Delivery DB и пропускает обработку.

Для защиты от дублирования при failover Delivery DB использует `SELECT FOR UPDATE` при смене состояния. Таким образом два экземпляра Delivery Engine не могут одновременно перевести одно уведомление из SENDING в FAILOVER.

Маркетинговые кампании: отдельный топик `mkt.notifications` с consumer group, ограниченной по throughput. Кампания разбивается на батчи по 10 000 уведомлений - это предотвращает влияние на транзакционные топики.
